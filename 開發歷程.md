## 一、專案是什麼、為什麼做

**創意軌道規劃器** 是給「創意軌道」機器人競賽用的**路線策略規劃工具**。

作者曾擔任這個競賽的指導老師,發現學生最常卡住的不是動手拼,而是「**規劃**」這一步——手上就那幾片板子要整場循環使用,還得繞過所有加分點、最後超過終點線,很多學生連一條可行路線都排不出來,挫折感很重。

因此做了這個網站,用**自動規劃演算法(AI)**在幾秒內找出一條可行、且組數盡量少的擺法,讓學生:

- 先看到一個可行解,建立空間規劃的直覺;
- 把卡關的時間省下來,專注在理解規則與創意;
- 老師備課、出題、驗證題目「有沒有解」時也能快速確認。

## 二、競賽規則(內建邏輯)

- 場地:9 列(A–I)× 36 欄 + 終點區。
- 每組抽 **4 片板子**,整場循環使用(放完同樣 4 片再來)。
- **緩衝**:剛放下的板子要先隔幾片才能再用(正常 2 片)。
- 可「壓板」:板子能疊在已放的板子上、走回頭路。
- 軌道必須經過所有**加分點 ◎**;最後一片要超過終點線,且終點要**再多壓一塊板子**。
- 起點板放在起點區。

| 組別 | 抽板子 | S 板 | 加分點欄位 |
|------|--------|------|------------|
| 國小 | 4 片 | 無 | 第 12、24 欄 |
| 國中 | 4 片 | 每組 1 片(順序不限) | 第 9、18、27 欄 |
| 高中 | 4 片 | 每組 1 片(**必須是該組最後一片**) | 第 9、18、27 欄 |

> **S 板**是三向路口,算該回合的**第 5 片**,一回合只能放一次。

## 三、開發歷程(依需求順序)

### 階段 1：核心引擎與遊戲規則

- 建立 React + Vite 專案,實作板子幾何模型(`src/data/tiles.js`、`src/planner/geometry.mjs`):每片板子的佔格、連接埠、旋轉(只轉不鏡射)。
- 自寫 **A\* 自動規劃器**(`src/planner/solver.mjs`):以最少組數為目標,搜尋從起點到超過終點線、經過所有加分點的完整路線,支援壓板與緩衝規則。
- 三組別規則(國小 / 國中 / 高中)、抽籤 4 片循環、加分點抽列。
- **S 板**加入:三向路口,線從任一口進、另兩口擇一出;下方「使用組合」顯示成「**R進M出**」這類進出臂標籤,而不是只寫「S」。
- 終點「**多壓一塊板子**」收尾規則;規劃完成的**排列動畫**(由起點往終點逐片彈出)。

### 階段 2：互動與操作

- **手動輸入**:除了隨機抽籤,使用者可自己挑 4 片板子,並設定每個加分點的列(重現實際比賽題目)。
- **抽籤 / 手動輸入** 模式切換。
- **拖曳放置 + 吸附落點預覽**:拖板子到地圖時即時顯示會放到哪一格(綠=可放、紅=超出場地);**拖出地圖即自動移除**(因此移除了原本的橡皮擦按鈕);左鍵旋轉、右鍵移除。
- **激進模式**:把緩衝由 2 片放寬成 1 片,通常能找到更少組數的解。
- **S 板一回合限放一次**:手動放下 S 後立即鎖定,本回合無法再放第二片;「本輪」計數含 S 改為 /5(國中/高中)。

### 階段 3：介面演進(多次迭代)

介面經過數次重做,逐步從「工程測試風」走向「專業策略工具」:

1. **極簡扁平**:統一白底細框按鈕、淡色 chip,移除表情符號與雜訊。
2. **Apple Liquid Glass**:玻璃材質、毛玻璃模糊、漸層底色。
3. **改回扁平**(玻璃太花、不耐看)→ 回到乾淨扁平風。
4. **產品級 Dashboard**:Header + 卡片式控制台 + 路線地圖卡 + 策略路線 + Design Token。
5. **工具型專業介面(最終方向)**:刻意脫離「AI 模板感」——
   - **不對稱控制台**(設定窄欄 / 抽籤主欄 / 工具窄欄,單一面板加分隔線,不是三張等寬卡)。
   - **地圖當主角的 canvas**(輕量 toolbar + 內凹畫布,淡格線、精緻座標、成熟的起終點標示)。
   - **策略路線改成 step rail**(板子深色方塊、直放外框、S 出向橘色標籤)。
   - **板子工具盤改成資產盤**(分群、清楚的 hover/selected/已用狀態)。
   - 語意化用色(綠=啟用/成功、紅=終點/危險、橘=加分點/提醒)、克制的微互動。

同時處理:

- **自適應尺寸 / 響應式**:桌面三欄、平板單欄、地圖與策略列可水平捲動。
- **手機優化**:格子維持可讀的 24px(窄螢幕水平捲動而非縮到看不清)、按鈕加大到 44px、字級放大、控制台直堆。
- **穩定可讀地圖**:移除會讓地圖「一直縮小」的動態縮放,改成穩定 24–28px。
- 修正右側黑邊(底色畫在 html 根畫布,純色保底覆蓋整個視窗)。

### 階段 4：工程化與上線

- **生產建置 + 程式碼混淆**:`npm run build` 會 `vite build` 後再用 `javascript-obfuscator` 混淆自家程式碼(`scripts/obfuscate.mjs`),React 等第三方維持原樣。
- **GitHub 部署**:建立**私有** repo,GitHub Actions(`.github/workflows/deploy.yml`)在 push 到 `main` 時自動建置並發布到 **GitHub Pages**;repo 私有、網站公開可分享。
- **分頁圖示(favicon)**:內嵌 SVG(深色磚上的白色軌道,綠點起點、紅點終點)。
- 網頁標題改為 **創意軌道規劃**。
- 撰寫 **README**(專案緣由 + 競賽規則連結)與 **LICENSE**(保留版權、允許教育使用)。

## 四、重要決策與取捨

- **只混淆自家程式碼、不混淆 React**:混淆 React 只會肥大又拖慢,且它是公開原始碼無祕密可保護。
- **私有 repo + 公開網站**:別人只能用網址、看不到原始碼,網站上的 JS 也是混淆過的。
- **不用「縮小」解版面**:內容多時優先讓地圖水平捲動 / 整頁正常垂直捲動,維持元素可讀,而不是把地圖縮到很小。
- **工作流程約定**:之後每次改動**先在本地(localhost:5173)確認**,使用者說 OK 才 `git push` 部署。
